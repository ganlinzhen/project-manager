# 工作管理仓库规范

## 1. 定位

工作管理仓库是多个业务项目的控制台，不是任一业务项目的代码仓库。桌面应用从这里读取项目配置并写入任务工件；Codex 在这里读取项目约定并管理需求上下文。

## 2. 目录规范

```text
work-manager/
├── AGENTS.md
├── projects/
│   └── <项目 ID>.yaml
├── project-rules/
│   └── <项目 ID>.md
├── data/
│   └── artifacts/
│       └── <任务 ID>/
└── docs/
    ├── 工作管理仓库规范.md
    └── <主题>.md
```

| 目录或文件 | 职责 | 维护方式 |
| --- | --- | --- |
| `projects/` | 项目机器可读配置 | 每个项目一个 YAML |
| `project-rules/` | 分支、合并、部署和验证规则 | 与项目 ID 同名 |
| `data/artifacts/` | 任务需求、上下文、计划、进展和完成总结 | 由工作管理器生成和更新 |
| `docs/` | 仓库级规范、决策记录和流程说明 | 人工与 Codex 共同维护 |
| `AGENTS.md` | Codex 在本仓库内的总协作边界 | 仓库维护者维护 |

## 3. 项目配置与规则文档

项目 ID 使用小写字母、数字和短横线，例如 `project-a`。对应配置与规则文件必须成对出现：

```text
projects/project-a.yaml
project-rules/project-a.md
```

YAML 只保存结构化运行配置：仓库绝对路径、默认分支、Issue 提供方和开发服务。不要把分支策略、发布说明或长篇背景写进 YAML；这些内容放入同名 Markdown 规则文档。

## 4. 规则优先级

执行某项目的需求时，按以下优先级处理：

1. 用户在当前对话中的明确要求。
2. 仓库根目录 `AGENTS.md`。
3. `project-rules/<项目 ID>.md`。
4. `projects/<项目 ID>.yaml` 中的结构化配置。
5. 任务工件和历史决策记录。

规则冲突或缺失时，先说明冲突或缺失，再请求确认；不要猜测分支、合并或发布方式。

## 5. 任务工件

每个任务使用 `data/artifacts/<任务 ID>/` 目录，包含：

- `requirements.md`：需求摘要和验收标准；
- `context.md`：项目背景、约束和关联资源；
- `plan.md`：实施步骤；
- `progress.md`：当前进展与下一步；
- `completion.md`：验证结果、交付内容和后续事项。

任务状态以工作管理器数据库和 CLI 查询结果为准，Markdown 用于保存可阅读的上下文。

## 6. 新增真实项目

1. 创建 `projects/<项目 ID>.yaml`，填写真实的绝对 `repositoryPath`、`defaultBranch` 和可选服务。
2. 创建同名 `project-rules/<项目 ID>.md`，至少说明分支命名、合并前验证、部署方式、回滚方式和禁止事项。
3. 省略 `mode` 或设置 `mode: real`。
4. 运行 `wm project validate <项目 ID> --json`，确保仓库路径、Git、默认分支、Issue 和服务配置可用。
5. 通过桌面应用或 CLI 创建并推进任务。

## 7. Demo 项目

`mode: demo` 是唯一允许跳过外部校验的模式，仅用于展示。它必须使用明显不可用的绝对占位路径，并且不得连接真实代码仓库、Git、Issue 或开发服务。真实项目不得使用 `mode: demo`。
