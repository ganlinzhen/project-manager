# 工作管理仓库模板与 Demo 项目设计

## 目标

提供一份可复制的工作管理仓库模板。模板既是桌面应用的本地数据根目录，也是用户在 Codex 中用于管理需求、沉淀项目规则和恢复上下文的工作目录。

模板内置一个 `mode: demo` 的演示项目：无需真实代码仓库、Git 分支、Issue 登录态或本地服务，即可在桌面看板展示示例任务和任务详情。

## 非目标

- 不放宽任何真实项目的仓库、Git、Issue 或服务校验。
- 不为 Demo 创建真实 Git 仓库、worktree、分支、远程 Issue 或开发进程。
- 不改变真实项目的任务创建、工件和服务管理流程。

## 模板目录

模板存放于当前产品仓库的 `templates/work-manager/`，便于复制到用户选择的任意本地目录后作为“工作管理仓库”配置给桌面应用。

```text
templates/work-manager/
├── README.md
├── AGENTS.md
├── projects/
│   └── demo.yaml
├── project-rules/
│   └── demo.md
├── docs/
│   └── 工作管理仓库规范.md
└── data/
    └── artifacts/
        └── .gitkeep
```

`projects/*.yaml` 是工作管理器识别项目的唯一位置；`project-rules/*.md` 是 Codex 与人工协作时应读取的项目规则；`data/artifacts/` 保存任务自动生成的需求、上下文、计划、进展和完成总结。

## Codex 协作约定

模板根目录的 `AGENTS.md` 规定：当需求明确关联某项目时，先读取 `project-rules/<项目 ID>.md`；规则不存在时，先说明缺失并与用户确认，再执行会影响该项目的 Git、发布或外部操作。

项目规则文档至少包含仓库位置、默认分支、分支命名、合并要求、部署方式、验证命令和禁止事项。模板中的 `demo.md` 用示例值展示这些字段，且明确标记为非真实环境。

## Demo 项目模型

在 `ProjectConfig` 中增加可选字段 `mode`，允许值为 `real`（默认）和 `demo`。`mode: demo` 是唯一的校验豁免开关，配置加载时仍要求项目 ID、名称、任务前缀、绝对形式的 `repositoryPath` 和默认分支，以维持配置结构一致。

Demo 配置使用一个明确不可用但格式正确的绝对占位路径，例如 `/__work-manager-demo__/repository`；它不能被误当作真实项目路径。

## 运行行为

### 配置校验

`wm project validate <id>` 对真实项目维持现有检查：路径存在、位于 Git 工作树、默认分支存在、服务命令可执行、Issue 提供方可访问。

对 `mode: demo`，命令不执行任何外部检查，返回 `valid: true`，并在结果中标明该项目为演示模式和已跳过的检查项。

### 示例数据

运行时发现 demo 项目后，若本地数据库尚无该项目的任务，则写入固定的四条示例任务：需求澄清、开发中、等待依赖和已完成。每条任务都包含优先级、当前进展、下一步和完整的 Markdown 工件，以便桌面看板和详情页都能展示真实数据结构。

种子仅按项目 ID 在当前数据库中初始化一次，不覆盖用户对 demo 任务状态或工件的修改，也不会为任何 `mode: real` 项目写入数据。

### 外部操作边界

Demo 任务保留纯本地操作：查看、更新进展、暂停、恢复、完成、重开和读取工件。

以下操作对 demo 项目必须拒绝并返回明确的稳定错误：创建或重试 worktree、启动或停止开发服务、创建或关联 Issue。桌面端根据 `mode` 隐藏或禁用相应入口，并说明“演示项目不连接真实仓库”。

## 数据流

```text
templates/work-manager/projects/demo.yaml
        │
        ▼
配置加载（mode: demo） ──► Demo 种子初始化 ──► SQLite 与 data/artifacts/
        │                                              │
        └──► 校验跳过外部依赖                            ▼
                                             CLI 与桌面看板/详情页
```

## 错误处理

- 非 `demo` 模式仍严格执行现有校验，避免错误配置被掩盖。
- Demo 外部操作的错误消息须说明操作被演示模式禁止，不把缺失路径或 Git 错误暴露为误导性故障。
- 种子创建失败时，运行时应返回可读错误；不可生成部分任务后静默继续。

## 验证

- 单元测试覆盖 `mode` 的默认值、demo 配置加载和真实配置仍受绝对路径校验约束。
- CLI 测试覆盖 demo 项目校验成功且不调用 Git/Issue/服务检查。
- CLI 或 Core 测试覆盖首次初始化的四条示例任务和工件；第二次初始化不重复写入。
- 桌面 UI 测试覆盖 demo 项目的演示标识与外部操作禁用提示。
- 使用临时工作管理根目录运行 CLI，确认模板的 `demo.yaml` 可加载且看板读取到示例数据。
