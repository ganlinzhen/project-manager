# Codex 项目目录初始化设计

## 目标

在桌面设置页中引导用户创建一个可直接被工作管理器使用的 Codex 项目目录。用户填写项目文件夹名称并选择本机父目录后，应用在目标位置创建新文件夹、复制内置 `templates/work-manager` 模板、保存为当前工作管理目录，并立即刷新项目和任务数据。

## 界面

- 将设置页中原“工作管理仓库”的区块标题改为“Codex 项目目录”。
- 未配置目录时，区块内显示“初始化项目”按钮。
- 已配置目录时，按钮变为“清空并重新设置”。
- 初始化弹窗包含两个必填项：项目名称与本机目录位置。目录项通过桌面原生目录选择器完成，而非手动输入。

## 初始化流程

1. 用户输入项目名称并选择父目录。
2. 主进程验证项目名称不为空、不含路径分隔符，父目录存在且可写。
3. 主进程计算目标目录为 `<父目录>/<项目名称>`；若该路径已存在，返回明确错误，不覆盖任何已有内容。
4. 主进程创建目标目录，并完整复制打包资源中的 `templates/work-manager` 内容。
5. 主进程将目标目录保存为 `managerRoot`，保留当前 Node.js 配置。
6. 前端更新设置状态并重新读取项目、任务列表；模板中的 demo 项目会由现有 CLI 初始化到 SQLite。

## 清空并重新设置

“清空并重新设置”会删除当前工作管理器对应的本地 SQLite 数据，清空保存的 `managerRoot`，然后打开同一初始化弹窗。它不会删除用户在文件系统中已创建的 Codex 项目目录或其文件，以避免误删用户内容。

重新初始化后，应用使用新目录重新加载项目配置和任务；旧数据库中的项目、任务、事件和工件索引均不再展示。

## 架构与安全边界

- React 设置页只管理表单、弹窗、加载态和反馈提示。
- `wmApi` 暴露初始化、重设和原生目录选择三个明确 API。
- Tauri Rust 主进程负责原生目录选择、路径校验、模板复制、设置持久化与 SQLite 清理。
- 模板必须作为 Tauri 资源随应用打包，开发环境也从产品仓库中的模板路径读取。
- 不向网页层开放通用文件读写能力。

## 错误处理

- 用户取消目录选择：保持弹窗打开，不提示错误。
- 名称为空、含路径分隔符或目标已存在：在弹窗内给出明确原因。
- 模板复制或设置保存失败：返回失败并保留原有配置；若创建了空目标目录则仅清理由本次操作新建且仍为空的目录。
- 清空数据库失败：保持现有配置，不进入重新设置状态。

## 验证

- Rust 单元测试覆盖目标路径校验、同名目录拒绝、模板复制和重设时清理 SQLite。
- 桌面 UI 测试覆盖新标题、未初始化状态的初始化按钮、初始化表单提交及成功后的按钮切换。
- 执行桌面端测试、类型检查和构建；在 Tauri 开发环境中实际选择临时目录，确认模板生成、demo 项目和任务刷新可见。

## 非目标

- 不删除用户创建的项目目录。
- 不覆盖同名目录。
- 不改变现有真实项目的 CLI 校验规则、任务模型或外部服务行为。
